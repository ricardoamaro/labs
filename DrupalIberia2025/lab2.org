* Understanding the Kubernetes Cluster Setup in Lab1

The use of =--servers 1= and =--agents 2= flags in the =k3d cluster create= command specifies the architecture
of the Kubernetes cluster being created with k3d. Below is a detailed explanation of what each flag signifies
and why such a configuration might be chosen.

** --servers 1
This flag instructs k3d to create the cluster with 1 server node. In the context of Kubernetes:
- A server node acts as a master node.
- It is responsible for managing the cluster and scheduling applications to run on agent nodes.
- A cluster requires at least one server node to function, but more can be added for high availability.

** --agents 2
By setting this flag, the cluster will include 2 agent nodes. In Kubernetes terms:
- Agent nodes, or worker nodes, are where applications (containers/pods) are actually executed.
- Having multiple agent nodes allows for the distribution of applications across more than one node, improving performance and reliability.

** Accessing nodes
A particular node can be accessed by attaching another pod running an image:
#+begin_src bash
kubectl debug node/k3d-drupal-cluster-agent-0 -it --image=ubuntu
#+end_src

** Why Use This Configuration?
*** Load Distribution
- Having multiple agent nodes allows applications to be spread out across nodes, enhancing performance and fault tolerance.
*** Resource Allocation
- More nodes mean more compute resources (CPU, memory) are available for applications, accommodating more demanding workloads or multiple applications.
*** Learning and Experimentation
- A multi-node setup is beneficial for understanding how Kubernetes manages resources and schedules workloads, offering a closer representation of a production environment compared to a single-node setup.
*** Availability
- While this minimal setup doesn’t achieve full high availability (since there's only one server node), it introduces the concept. In production, multiple server nodes are recommended to ensure operational continuity in case of a node failure.
Therefore, the =--servers 1 --agents 2= configuration creates a small, but more realistic, multi-node Kubernetes cluster.

* Use namespaces to separate Drupal sites



** Create a namespace for the Drupal deployment:

- What is a namespace?
In Kubernetes, a **namespace** partitions cluster resources for multiple users or teams,
ensuring organization and management. It offers isolation, resource quotas, and access control.
While not a strict security boundary, namespaces help logically separate resources.
They are essential for large environments, allowing efficient, organized, and controlled resource management.

#+begin_src bash

* Increase the number of replicas of the drupal deployment to 2

** Method 1: Using kubectl scale
This command tells Kubernetes to adjust the my-drupal deployment to have 2 replicas:
#+begin_src bash
kubectl scale deployment my-drupal --replicas=2 -n drupal
#+end_src

** Method 2: Editing the Deployment

Alternatively, you can manually edit the deployment to change the number of replicas.
#+begin_src bash
kubectl edit deployment my-drupal -n drupal
#+end_src
This command opens the deployment's YAML in your default text editor. Find the replicas field under spec and change its value from 1 to 2. If the replicas field is not present, you can add it under spec:
#+begin_src yaml
spec:
replicas: 2
#+end_src
After saving your changes and exiting the editor, Kubernetes applies the update the deployment, scaling it to 2 replicas.

** Method 3: Installing with a values.yaml file

1. Download and modify the default values file:
#+BEGIN_SRC bash
curl -s https://raw.githubusercontent.com/bitnami/charts/refs/heads/main/bitnami/drupal/values.yaml
#+END_SRC
2. Start the my-drupal deployment with the replicas changes in the values file
#+BEGIN_SRC bash
helm install my-drupal bitnami/drupal -f values.yaml -n drupal
#+END_SRC

    Example:
    #+begin_src yaml
    drupal:
      resources:
        requests:

** CI/CD Pipeline (Basic)
1. Set up a simple CI/CD pipeline using a tool like JenkinsX or GitLab CI to automate the process of building a Drupal image, pushing it to a container registry, and deploying it to the Kubernetes cluster. (Detailed steps depend on the chosen CI/CD tool).


*** Prerequisites

1. **Kubernetes Cluster**: Ensure you have access to a running Kubernetes cluster.
2. **Helm**: Installed and configured to manage your Kubernetes applications.
3. **Container Registry**: A Docker registry (e.g., Docker Hub, Google Container Registry, etc.) to store your Docker images.
4. **Jenkins**: Installed and running, with access to your source code repository and Kubernetes cluster.

*** Step-by-Step Setup

**** Step 1: Install Jenkins

1. **Deploy Jenkins to Kubernetes** (if not already set up):
   #+begin_src bash
   helm repo add jenkins https://charts.jenkins.io
   helm repo update
   helm install jenkins jenkins/jenkins --set controller.serviceType=LoadBalancer
   #+end_src

2. **Access Jenkins**: Obtain the admin password and access Jenkins UI.
   #+begin_src bash
   kubectl get svc --namespace default -w jenkins
   kubectl exec --namespace default -it svc/jenkins -c jenkins -- /bin/cat /run/secrets/additional/chart-admin-password && echo
   #+end_src

**** Step 2: Create a Jenkins Pipeline

1. **Install Required Plugins**: Install plugins for Docker, Kubernetes, Git, and any other necessary integrations.

2. **Create a New Pipeline Job**: In Jenkins, create a new pipeline job.

3. **Configure Jenkinsfile**: Add a `Jenkinsfile` to your repository. Here's a basic example:

   #+begin_src groovy
   pipeline {
       agent any
       environment {
           DOCKER_REGISTRY_CREDENTIALS = credentials('docker-registry-credentials-id')
           KUBECONFIG_CREDENTIALS = credentials('kubeconfig-credentials-id')
           DOCKER_IMAGE = "your-docker-repo/drupal:${env.BUILD_NUMBER}"
       }
       stages {
           stage('Build') {
               steps {
                   script {
                       docker.build(DOCKER_IMAGE)
                   }
               }
           }
           stage('Push') {
               steps {
                   script {
                       docker.withRegistry('https://index.docker.io/v1/', DOCKER_REGISTRY_CREDENTIALS) {
                           docker.image(DOCKER_IMAGE).push()
                       }
                   }
               }
           }
           stage('Deploy') {
               steps {
                   script {
                       withKubeConfig([credentialsId: 'KUBECONFIG_CREDENTIALS']) {
                           sh "helm upgrade --install my-drupal bitnami/drupal --set image.repository=${DOCKER_IMAGE}"
                       }
                   }
               }
           }
       }
   }
   #+end_src

   - **Environment Variables**: Adjust `DOCKER_IMAGE`, `DOCKER_REGISTRY_CREDENTIALS`, and `KUBECONFIG_CREDENTIALS` as needed.
   - **Credentials**: Ensure Jenkins has the necessary credentials configured for Docker registry and Kubernetes access.

**** Step 3: Configure Jenkins

1. **Credentials**:
   - Add Docker registry credentials in Jenkins (`docker-registry-credentials-id`).
   - Add Kubernetes credentials (Kubeconfig or token) in Jenkins (`kubeconfig-credentials-id`).

2. **Source Code Management**: Connect Jenkins to your source code repository (e.g., GitHub, GitLab).

3. **Trigger Builds**: Set up webhooks or polling to trigger the Jenkins job on code changes.



** Configuration Management

A ConfigMap is a Kubernetes resource used to store configuration data in key-value pairs. It allows you to decouple configuration artifacts from image content to keep containerized applications portable. ConfigMaps are useful for managing application configuration without the need to rebuild container images or hard-code configuration data within the application itself.

*** Step 1: Create a ConfigMap

First, create a ConfigMap with the desired configuration settings. For example, you might want to set the site name or other Drupal-specific settings.

    #+begin_src bash
    kubectl create configmap drupal-config --from-literal=DRUPAL_SITE_NAME="My Drupal Site" -n default
    #+end_src

This command creates a ConfigMap named drupal-config in the default namespace with a key-value pair for DRUPAL_SITE_NAME.

*** Step 2: Reference the ConfigMap in the Helm Chart

To use the ConfigMap in the Bitnami Drupal chart, you need to modify the values.yaml file to include the configuration as environment variables or mounted files.
Using Environment Variables

1. Edit your values.yaml: Add an extraEnvVars section to include environment variables from the ConfigMap:

    #+begin_src yaml
    extraEnvVars:
      - name: DRUPAL_SITE_NAME
        valueFrom:
          configMapKeyRef:
            name: drupal-config
            key: DRUPAL_SITE_NAME
    #+end_src

2. Deploy or Update the Helm Release:

Use Helm to deploy the chart with the updated values:

    #+begin_src bash
    helm upgrade --install my-drupal bitnami/drupal -f values.yaml --namespace default
    #+end_src

**Using ConfigMap as a Volume**

Alternatively, you can mount the ConfigMap as a volume if you need to provide configuration files to Drupal:

1. Edit your values.yaml: Add a volumeMounts and volumes section:

    #+begin_src yaml
    volumeMounts:
      - name: config-volume
        mountPath: /etc/drupal-config

    volumes:
      - name: config-volume
        configMap:
          name: drupal-config
    #+end_src

2. Deploy or Update the Helm Release:

    Apply the changes with Helm:

    #+begin_src bash
    helm upgrade --install my-drupal bitnami/drupal -f values.yaml --namespace default
    #+end_src
